# 验证规则

## 必须执行的验证命令

每次代码修改后都要运行：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/agent-validate.ps1
```

## 停止条件

只有以下条件全部满足，任务才算完成：

- 请求的功能或修复已经生效
- 所有可用的自动验证检查通过
- 修改文件范围仅限当前任务
- 没有新增控制台、构建、lint、类型或测试错误
- 如果行为或架构发生变化，相关文档已经更新

## 失败处理闭环

验证失败时：

1. 阅读第一个有意义的错误，不要只看最后一行。
2. 判断失败是否由当前改动引起。
3. 修复最小根因。
4. 优先重新运行失败的单个命令。
5. 单个失败命令通过后，再运行完整验证命令。

## UI 改动验证

如果改动涉及 UI，还要验证：

- 桌面端布局
- 移动端布局
- 核心用户流程
- loading 状态
- empty 状态
- error 状态
- success 状态
- 文本没有重叠
- 滚动没有异常
- 浏览器控制台没有新增错误

如果当前环境无法做浏览器自动化验证，最终报告中必须明确说明“未运行 UI 浏览器验证”。

## Bug 修复规则

修 bug 时：

1. 先用失败测试或明确手动步骤复现问题。
2. 做最小修复。
3. 验证原失败测试或手动步骤已经通过。
4. 运行完整验证命令。

## 功能开发规则

开发功能时：

1. 只实现验收标准范围内的内容。
2. 在合理范围内新增或更新验证。
3. 逐条验证所有验收标准。
4. 运行完整验证命令。

## 禁止事项

- 不要因为改动看起来简单就跳过检查。
- 不要用“应该可以了”声明成功。
- 不要修复无关失败，除非它阻塞验证且确实必须处理。
- 不要为了绕开一个小错误而重写大段代码。
